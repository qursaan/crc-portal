1        Introduction

The purpose of teaching portal is to provide a framework for instructor to manage students’ reservation and experiments on testbeds. This portal is integrating with our main portal application which included scheduling, disk imaging, OMF manager, and other functionality. The teaching portal interface will simplify functions of registration, reservation, and experimentation test for both instructor and students

2        Registration

The registration is the basic steps; a new feature is added for determine the type of user during registration. Now, there are three type of users are included in the portal system:

1.      Researcher: The researcher user has all system functionality but excluded from teaching panel

2.      Instructor: The instructor user has all system functionality and additional feature of teaching portal such as bulk reservation, course setup, and control students.

3.      Student: The student user excludes from free reservation and limited function related to testbeds controller.

 

2.1      Create Instructor Account

To create new instructor account, from login page and click on “signup” button. (Figure 1) show the registration form. In the “Register as” field select “Instructor”

Figure 1: Create instructor account

 

Further, the admission of authority is necessary to accept the user registration. Once a user gets admission, a user account is created in our system, and a simple notification email is send to user’s email.

2.2      Create Student Account

To create new student account, from login page and click on “signup” button. (Figure 2) show the registration form. In the “Register as” field select “Student”, additional field appears to choose the supervisor name from a list of all instructor in system.

Figure 2: Create student account

Once student is registered, a notification will inform corresponding instructor to approve the user registration.

3.5      Create Experiments

Instructor can add experiments corresponding to course. In “Experiments list” page, there is a list for all experiments created by instructors divided into two sections, the active and expired experiments. The experiments list included a short detail for each (Figure 11).

To add new Experiments, click on “Add new experiment” button in “Experiments List” page.

In “New bulk reservation” page, Essential information about course and experiment (see Figure 12 and Figure 13) are required such as:

·         Experiment title: The experiment short description name.

·         Course Name: The course corresponding to this experiment.

·         Due Date: The date of expired the experiment.

·         Max students: The maximum number of student allowed to reserved slot for this experiment.

·         Reservation Type: There are two types which are Locked and Racing. Locked will prevent other system users from reserved the same bulk reservation type, while the Racing will allow other users to create reservation if slot is available even if the slot is mark for bulk student time. Till Now the Locked type is only active.

·         Control Options: The control option will allow instructor to predefined the functions available in experiment page for student.

The next details information is associated with reservation time and testbed type and hardware’s (see Figure 14), which is the same as normal researcher request () such as.

1.      Slice Name: The slice name for recorded his own slice in history.

2.      Specify the Servers/Machine Type: There are two types of servers, Simulation Server and Testbed Nodes

3.      Request Plan: The request plans are Normal/Scheduling request or Urgent request.

·         Normal Request: In this plan, the user has to request slice before actual reservation in one-day advance to choose from all available resources.

·         Urgent Request: In this plan, the user has to request slice in the current day and with only available resources.

4.      Slice Time: The user has to specify the reservation slice time need to running the test. The slice time should be chosen from current free timetable

5.      Determine the Hardware Specifications: The instructor determines the hardware specification of the servers and/or nodes need to be included in experiment

6.      Operating System Image: The instructor has to choose the image of the server in case choose a simulation server or a virtual machine. There are some default images will be available to all users. However, the user can set/upload an image from the Cloud disk

7.      The Test Purpose: The instructor can write his own purpose for history and logging reservation.

Once all information is set, instructor can press “Request Slice” and confirmation message will have displayed as well as reservation is saved. In the “Experiments List” page a new experiment will post in “Active Experiments” list. The instructor can cancel request at any time before the experiment expired date (see Figure 15).

4.5      Current Request

Upon the hardware is prepared and requested time slice is going, the scheduler service will push the request into the order upon the time of request is starting. The current request can be approved/pending based on scheduling criteria. In case of approving the request using automated system, the request will be listed in the “Active Process” (see Figure 23).

As well as the slice time is started, the action of current process will be active and user can access “Control Panel” by click on “Control” button (see Figure 24).
